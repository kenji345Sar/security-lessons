## TLS (Transport Layer Security)

2台のコンピュータが、**盗み見られず・なりすまされず** に通信するためのプロトコル。

### 名前の整理

| 呼び方 | 関係 |
|---|---|
| SSL | TLSの古い名前。今は非推奨だが「SSL証明書」のように慣用で残っている |
| TLS | 現行の正式名 (TLS 1.2 / 1.3 が現役) |
| HTTPS | HTTP の上に TLS を被せたもの。HTTP + TLS = HTTPS |

HTTPS は独立したプロトコルではなく、**HTTP を TLS の中に通している**だけ。

### やっていること (2つ)

1. **暗号化** — 通信内容を、当事者以外には読めないバイト列にする
2. **身元確認** — 相手が本物のサーバーかを **証明書** で確かめる

両方セットで初めて意味がある。暗号化だけだと「偽サーバーと暗号通信して情報を渡す」状態になる。

### TLSハンドシェイク (ざっくり)

```
クライアント                       サーバー
   |  「TLSで話そう。使える暗号方式は…」 →   |
   |   ← 「これで話そう。あと俺の証明書はこれ」 |
   | (証明書を検証: 本物のサーバーか?)         |
   |  鍵を交換 (お互いだけが知る共通鍵)        |
   |  ←  以降このセッションは暗号化通信  →    |
```

## 証明書 / CA (Certificate Authority)

サーバーが「私は ○○.com です」と名乗るときに提示する身分証。
**CA (認証局)** が署名したものなら、ブラウザ/OSは信頼する。

- 公的CA: Let's Encrypt, DigiCert など。ブラウザ/OSが最初から信頼リストを持っている。
- 自己署名証明書: 自分で署名した証明書。CAの署名がないので、クライアント側で「これを信用する」と明示しないとエラーになる (本リポジトリの `fixed-client.js` で `ca:` を渡しているのはこのため)。

### ブラウザの鍵マークの判定

ブラウザは以下を全部チェックする。1つでもダメだと警告画面。

1. サーバーが提示した証明書がCAに署名されているか
2. 証明書のドメイン (CN / SAN) と接続先ドメインが一致するか
3. 有効期限内か

## MITM (Man-In-The-Middle / 中間者攻撃)

クライアントとサーバーの間に攻撃者が割り込み、両者になりすます攻撃。

- HTTP の場合: 中身が平文なのでそのまま盗聴・改ざんできる。
- HTTPS で証明書検証を切っている場合: 攻撃者の偽証明書を受け入れてしまい、暗号化していても中身が抜かれる。

防御は **TLS + 証明書検証を切らないこと**。
