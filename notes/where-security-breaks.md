# セキュリティはどこで破られるのか

## 結論

- **HTTPS (TLS) を正しく使えば、経路の盗聴・改ざんは常識的に破られない。**
  - TLS 1.2 / 1.3 の暗号自体が解読された事例は (実務的には) ない。
- **破られるとしたら、TLS の外側 (他の層、または TLS の設定ミス)。**

つまり「HTTPS にしたから安全」ではなく「**経路だけは守った**」。
他の層で password が露出すれば、HTTPS は無意味になる。

## TLS が「破られる」ように見える数少ない例

これは TLS の暗号が解かれているのではなく、**使い方が間違っている**。

- 古い TLS バージョン (1.0 / 1.1) や弱い暗号スイートを許可している
- 証明書の有効期限切れ
- クライアント側で証明書検証を切っている (`rejectUnauthorized: false`, `-k`, `verify=False`)
- 自己署名証明書を盲目的に信頼している
- HTTP/HTTPS が混在し、Cookie が HTTP 側で漏れる

## 「TLS の外側」で破られる例 (現実はほぼこちら)

| 場所 | 例 | HTTPS で防げる? |
|---|---|---|
| 入力時 (ブラウザ内) | XSSでフォームの入力を盗まれる | × |
| サーバー側受信直後 | サーバーログに `password=p@ss` を残す | × |
| サーバーのDB | パスワードを平文保存して、DB が漏洩 | × |
| 認証ロジック | SQLインジェクションでログイン突破 | × |
| 利用者 | 偽サイトに誘導されてパスワード入力 (フィッシング) | × |
| 運用 | 退職者アカウントが残っていて使われた | × |

## 全体像での位置づけ

| 層 | 脅威 | 対処 |
|---|---|---|
| ネットワーク | 盗聴・改ざん | **HTTPS** |
| アプリケーション | XSS, SQLi, CSRF | コードで対処 (02-app) |
| 認証 | 弱いパスワード, トークン漏洩 | 認証設計 (03-auth) |
| データ | DB漏洩 | 暗号化・アクセス制御 (05-data) |
| 人 | フィッシング, 内部不正 | 教育・運用 (06-physical-ops) |

各層は **独立して破られうる**。HTTPS は 1 層分しかカバーしない。
だから 1 層ずつ理解して塞いでいく、というのがこのリポジトリの方針。
