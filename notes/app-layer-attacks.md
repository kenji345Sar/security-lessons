# アプリケーション層攻撃の見取り図

`02-app/` で扱う攻撃 (XSS / CSRF / SQLi …) の共通構造をまとめる。
個別の手口は各ディレクトリの README へ。ここは「全体を俯瞰する地図」。

---

## どこで起きている攻撃か

OSI モデルで言うと、これらは全部 **Layer 7 (アプリケーション層)** = HTTP / HTML / JS / Cookie のレイヤー。
`01-network/` で扱った TLS (Layer 6) より上。

| 攻撃 | 悪用される機能 |
|---|---|
| **XSS** | ブラウザの「HTML に含まれる JS を実行する」機能 |
| **CSRF** | ブラウザの「送り先ドメインの Cookie を自動付与する」機能 |
| **SQLi** | サーバー側の「受け取った文字列を SQL として実行する」機能 |

---

## 「プロトコルのバグ」ではなく「仕様通り動いているのを悪用」

ここが押さえどころ。HTTP もブラウザも、攻撃時に **故障していない**。
仕様に書かれた通りに忠実に動いている。

| 仕様 | 悪用される側面 |
|---|---|
| サーバーが返した HTML の `<script>` はブラウザが実行する | 攻撃者の `<script>` が混入していても忠実に実行 |
| 指定ドメイン宛のリクエストに Cookie を自動付与 | 攻撃者ページから飛んだリクエストにも忠実に付ける |
| フォームは別オリジンにも送信できる | 攻撃者ページから標的サイトに POST 可能 |
| サーバーは届いた文字列を解釈して動く | SQL の中に `' OR 1=1 --` が混入していても忠実に解釈 |

これらは 1994 年〜2000 年代初頭に決まったルール。当時は
「悪意あるサイトが他サイトを攻撃する」シナリオを十分に考慮していなかった。
今からプロトコルを書き直すと既存サイトが全部壊れるので、**追加のルールを後付けで載せて穴を塞いでいる**のが現状。

### 後付けの代表

| 後付け | 何を塞ぐか |
|---|---|
| CSP (Content-Security-Policy) | XSS の影響範囲を制限 (inline script 禁止など) |
| SameSite Cookie | CSRF を成立しにくくする (他オリジンからの Cookie 自動付与を止める) |
| CORS | クロスオリジン JS の読み書きを制限 |
| X-Frame-Options / frame-ancestors | クリックジャッキング対策 |
| HSTS | HTTP への降格を防ぐ |

---

## だから「アプリ側で対策する」が必要

HTTP もブラウザも **悪意の有無を判断しない**。

- サーバーに届いた `<script>` が「攻撃」か「普通の投稿」か → HTTP は区別できない
- 送られてきた Cookie が「本人意思」か「CSRF」か → HTTP は区別できない
- 受け取った文字列が「ユーザー入力」か「SQL文の一部」か → DB は区別できない

→ **アプリ側のコードが追加で確認する**しかない。これが weak/fixed の比較で見ていること。

| 攻撃 | アプリ側で何を確認するか (≒ fixed の追加分) |
|---|---|
| XSS | 出力時に HTML エスケープする |
| CSRF | CSRF トークンを検証する / SameSite Cookie を使う |
| SQLi | プレースホルダで SQL を組む (文字列連結しない) |

---

## まとめ (ひとこと)

> **「ブラウザ/HTTP の挙動を、本来の目的とは違う組み合わせで使われると悪さができてしまう」**
>
> プロトコル自体は壊れていない。**仕様の組み合わせに穴がある**。
> その穴は後付けの追加ルール (CSP, SameSite, CORS …) と
> アプリ側のコード対策 (エスケープ、トークン、プレースホルダ) で塞ぐ。
