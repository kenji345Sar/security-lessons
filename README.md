# security-lessons

JavaScript でセキュリティを 1 層ずつコード例で学ぶ。

## 全体像

「セキュリティ」は単語が広すぎる。**何を / 何から守るか** で層を分けて扱う。

| 層 | 守る対象 | 主な脅威 | 対応策の例 |
|---|---|---|---|
| ネットワーク | 通信内容 | 盗聴、改ざん | TLS、VPN、ファイアウォール |
| OS / エンドポイント | 端末そのもの | マルウェア、不正侵入 | ウイルススキャン、EDR、パッチ適用 |
| アプリケーション | Web アプリ | XSS、SQLi、CSRF | 入力検証、WAF |
| 認証・認可 | ユーザー識別 | なりすまし、権限昇格 | MFA、RBAC、トークン管理 |
| データ | 保存中のデータ | 漏洩、持ち出し | 暗号化、アクセス制御、DLP |
| 物理 | 機器・媒体 | 盗難、USB 持込 | 入退室管理、媒体管理 |
| 運用・人 | 組織全体 | 誤操作、内部不正 | 監査ログ、教育、権限分離 |

## 進め方

1 層につき以下の 3 点セットで進める。

1. **弱いコード** — 脆弱な状態の最小サンプル
2. **攻撃される様子** — どう悪用されるか
3. **直したコード** — 修正後のサンプル

## 言語

JavaScript (Node.js / ブラウザ)。

## 実行環境（重要な前提）

このプロジェクトのコード例は、**Mac の上で Node プロセスとして** サーバとクライアントを起動して進める。

- 「サーバ」 = `node weak-server.js` で起動した **Node プロセス**（Mac の中の 1 プロセス）
- 「クライアント」 = ブラウザや `node weak-client.js`（同じ Mac の中の別プロセス）
- 「ネットワーク」 = ループバック (`localhost` / `127.0.0.1` / `lo0`)
- OS は **1 台の Mac の macOS だけ**。別の OS や別マシンは出てこない。

### なぜこの簡略化で成立するか

XSS / CSRF / SQLi のような **Web アプリの脆弱性は、アプリのコードそのものに存在する** バグ。攻撃成立の条件は「攻撃者のブラウザから正規サーバへ HTTP リクエストが届くこと」だけで、サーバが別マシンにある必要はない。Mac の中の Node プロセスでも、実機サーバでも、再現される現象は同じ。

### この簡略化で見えなくなるもの

代わりに、以下は本プロジェクトでは扱えない:

- OS が証明書をどこに保存するか（`/etc/ssl/certs/` などの OS の信頼ストア）
- nginx などの **OS のサービス** が TLS を終端する流れ
- 別マシンの **OS そのものに対する攻撃**（マルウェア、カーネル脆弱性、パッチ管理）
- ロードバランサや配信網など、**マシン同士の構成** で守る考え方

これらは別プロジェクト [`infra-lessons`](../infra-lessons/) で、**実際に複数の Ubuntu VM を立てて** 扱う。レイアの違いは [`infra-lessons/notes/app-vs-infra.md`](../infra-lessons/notes/app-vs-infra.md) を参照。

### 例外: 04-endpoint

エンドポイント（OS）セキュリティは **そもそも Node プロセスでは再現できない** 領域なので、`04-endpoint/` だけは Node コードを書かず、概念と OS 付属ツール（macOS の `codesign` など）で進める。実機ハンズオンは `infra-lessons` の VM 環境を使う想定。

## 学習順序（予定）

1. ネットワーク層 (TLS / HTTPS)
2. アプリケーション層 (XSS, SQLi, CSRF)
3. 認証・認可 (パスワード保存, JWT)
4. エンドポイント (ウイルススキャン) ※概念中心
5. データ層 (暗号化、アクセス制御)
6. 物理・運用層 ※概念中心
